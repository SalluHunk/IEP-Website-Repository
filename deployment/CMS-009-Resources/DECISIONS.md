# CMS-009 — Resources Module: Decisions

## No real content exists — confirmed before building anything, not assumed

Before writing this package, checked: `wp_list_pages` for the live Resources page (984, found — 100% "Coming soon" placeholder copy for all 3 categories); `wp_search`/`wp_search_posts` for "download"/"resource"; every governance-repository document (`KNOWLEDGE-SOURCES-001`, `CRN-001`, `PDC-001`, all of `content/CONTENT-001-Services/`) grepped for "resource|whitepaper|calculator|funding briefing" — no real Resources content anywhere. This is a genuine missing-client-content situation, distinct from every prior CMS module (WP-03–06 all had real content already live to migrate). Presented directly to Mission Control as a fork rather than guessed at: build infrastructure only / defer the whole Work Package / investigate the Grants page first / "I have real content" — Mission Control chose **build infrastructure only**.

## Fresh CPT (`cspt-resource`), not native Media

The original Blocker A scoping note flagged this as worth checking explicitly: WordPress's native Media (attachment) type is already REST-exposed, so a Resources listing built on Posts/Pages + Media was a real alternative to a new CPT. Chose a CPT anyway, for the same reason every other module needed one: attachments don't natively support custom taxonomy-like categorisation (3 fixed categories), a `gated` flag, or a `display_order` without significant extra plumbing on top of the Media Library — plumbing that would end up re-inventing most of a CPT's normal capabilities anyway, and would tangle "downloadable resource" metadata onto the same object type used for every image on the site. A CPT keeps this module's data model isolated and consistent with Leadership/Case Studies/Industries/Testimonials, at negligible extra cost (this is a proven, fast pattern by the 5th time). The CPT's File field still stores an actual Media Library attachment underneath (ACF's `file` field type returns an attachment array) — so this isn't duplicating the Media Library, just wrapping it with the metadata attachments alone can't carry.

## Grid only, no individual detail pages — made directly, not asked, unlike WP-05/06

WP-05 and WP-06 both surfaced the "individual pages vs. grid" fork to Mission Control explicitly, because in both cases there was a real product-shape question (Industries could plausibly want a landing page per sector; testimonials could plausibly want to be read individually). Here the shape is unambiguous enough not to warrant another question: a downloadable resource has no narrative content of its own — a visitor's only actions are "download" or "don't." There's no reading experience a detail page could offer that the card itself doesn't already provide. Documented here rather than asked, so a future session understands this was a judgment call, not an oversight.

## `gated` field ships as a flag only — no MC4WP enforcement built

Page 984's own copy says gated downloads are "to be wired to Mailchimp (MC4WP) when assets are ready." `CRN-001` separately lists "newsletter implementation" as an explicitly open decision. Building real email-gate enforcement now would mean guessing at both what MC4WP integration approach Mission Control wants *and* whether gating applies per-resource or site-wide — two undecided things, not one. The field exists so the eventual decision has somewhere to attach without a schema migration, but `[iep_resource_grid]` deliberately does not expose a working download link for any resource flagged `gated` (renders "Registration required — coming soon" instead) — exposing the raw file URL would silently defeat the point of the flag being there at all, which would be worse than not building the flag.

## 3 categories are fixed, taken verbatim from existing page copy

"Guides & whitepapers" / "Tools & calculators" / "Funding briefings" are not invented — they're the exact category labels already live on page 984's "Coming soon" cards, chosen by whoever built that page in an earlier session. Reusing them rather than proposing new ones keeps this package non-disruptive to a decision that already appears to have been made once (the page's own information architecture), even though no content exists to populate it yet.

## The Grants page (648) — flagged as a possible future seed, not touched

Found while searching for existing Resources-adjacent content: a "Grants" page (648, real content about IETF grant funding, live and published) exists separately from the Resources page, is not linked from the main navigation, and is dated 2025-03 — pre-dating this entire redesign engagement, authored by a different user (`klaudiadev24`, not the current session's author `sidgaur`), and containing a stale application deadline (19 April 2024, already passed). It's a plausible real seed for the "Funding briefings" category — genuine primary-source content, not fabricated — but publishing anything derived from it without the client confirming the figures/deadlines are still current would risk shipping outdated funding information, which is worse than shipping nothing. Not merged into this package. Flagged for Mission Control's future call, consistent with the repository's "flag, don't silently resolve" discipline.

## Resources page not in main navigation — noted, not changed

`wp_list_menu_items` on the Main Menu (id 27) confirmed 9 items (Home/About/Services/Industries/Case Studies/Leadership/Testimonials/Insights/Contact) — no Resources link. Could be intentional (soft-launched ahead of real content, matching its own "Coming soon" state) or an oversight from whichever earlier session created the page. This package doesn't add it to the nav — doing so would draw visitor traffic toward a page that currently offers nothing to download, which seems actively counterproductive while the module ships with zero content. Worth revisiting once real resources exist.

## Icon fix (fa-file-lines, fa-hand-holding-dollar) — fixed live, not part of the package

While verifying the 3 existing category icons wouldn't be silently broken if reused as reference points for this package's own visual language, found (4th instance of this exact recurring bug on this site — previously hit in CMS-005A, WP-04's Case Studies deployment, and WP-05's Industries page): `fa-file-lines` and `fa-hand-holding-dollar` are both FA6-only names, invalid on this site's actual bundled Font Awesome 5.15.3, rendering as invisible icons (`content: none`, confirmed via live `getComputedStyle(el, '::before')` inspection, not assumed). `fa-calculator` was already valid. Verified FA5-correct replacements the same empirical way before applying (`fa-file-alt`, codepoint `f15c`; `fa-hand-holding-usd`, codepoint `f4c0`) and fixed directly on page 984's live `_elementor_data`, independently re-verified after the write. This is a live production fix, same category as WP-05's page-971 icon fixes — not part of the CMS-009 package itself, since it touches existing page markup rather than new infrastructure.
