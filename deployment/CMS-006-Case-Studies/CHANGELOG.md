# CMS-006 — Case Studies Module: Changelog

- Initial deployment package. Repurposes the existing `cspt-portfolio` CPT (Greenly theme, currently 12 unrelated Envato demo entries) as the Case Studies CMS module, per the live Case Studies page's (978) own embedded developer note naming this CPT as the intended target.
- ACF field group: Icon, Summary Snapshot, Challenge, Solution, Results, Commercial Impact, Display Order (7 fields). No Client or Address fields — see `DECISIONS.md`.
- Sector taxonomy: retrofits REST support onto the existing `portfolio-category` taxonomy, repurposed with 3 new terms (Food & Beverage, Manufacturing, Construction Materials) drawn directly from the content source document's own sector groupings — the demo terms (Solar/Sun/Energy) are not reused.
- PHP Code Snippet: REST retrofit for CPT + taxonomy, authenticated `/iep-cms/v1/case-study/{id}` read/write endpoint (+ `/use-single-shortcode` and `/unpublish` helper routes), `[iep_case_study_grid]` and `[iep_case_study_single]` shortcodes.
- Full migration table for all 6 real case studies (Plastic Packaging Manufacturer, Urban Brewery, Aluminium Recycling Plant, Specialist Brick Manufacturing, Brewer's Spent Grain, Net-Zero Brewery Design), sourced from the live, already-approved Case Studies page copy plus the Primary Source content document's Challenge/Solution/Results narrative.
- Not yet applied to WordPress — package phase only. See `missions/WCP-001-Progress-Register.md` for WP-04's execution report.

## v1.1 (2026-07-18, same day) — deployed to production
- User installed the ACF field group and Code Snippet; deployment proceeded through 3 further live-diagnosed fixes (all folded into `php/case-study-helper-functions.php`, not left as separate patches):
  1. **Real sector taxonomy key discovered dynamically.** The guessed key `portfolio-category` (from the front-end URL) doesn't match the theme's actual registration; real key is `cspt-portfolio-category`. Part 2 now discovers it at runtime via `get_object_taxonomies()` instead of hardcoding a guess, storing it in an option all other parts read from.
  2. **`wp_set_object_terms()` int-cast bug found and fixed.** The sector-assignment endpoint passed a numeric string (e.g. `"39"`) instead of a real PHP `int`; `wp_set_object_terms()` only treats `is_int()` values as term IDs, so the string was treated as a term *name* search and silently created 3 bogus junk terms (literally named "38"/"39"/"40", term IDs 41–43). Fixed by casting to `(int)`; the 3 junk terms were deleted after the fix, and all 6 real case studies were re-assigned to the correct terms (38/39/40).
  3. **Theme's hardcoded single-post template bypassed.** Unlike CMS-002's Elementor-override issue, this CPT's `single-cspt-portfolio.php` renders a hardcoded "About the project" Date/Client/Category/Address demo box directly in PHP, independent of `post_content` — confirmed live, since it persisted even after `post_content` was fully replaced with `[iep_case_study_single]`. Fixed with a new `template_redirect` short-circuit (Part 3b) that keeps the theme's header/footer but bypasses its hardcoded body markup for `is_singular('cspt-portfolio')`.
- `featured_media` clearing was also missing from the first draft of the endpoint (present in CMS-002's pattern, omitted here by oversight) — added before the first content POST reached the live posts.
- All 6 real case studies fully populated, featured images cleared, sectors assigned, single-page shortcode redirect applied; 6 surplus demo posts moved to draft. Verified live via cache-busted fetch (`fetch(..., {cache:'no-store'})`) after each fix — host-level page cache (GoDaddy) served stale/empty responses at several points during diagnosis, consistent with this site's established caching behavior, not a data or code problem each time it happened.
- See `missions/WCP-001-Progress-Register.md` for the full WP-04 deployment report.

## v1.2 (2026-07-18, same day) — content-fidelity audit + card-to-detail linking
User asked to confirm (1) all client-provided case study content was actually captured, and (2) whether the live Case Studies page (978) cards actually link to their detail pages. Both checked directly against source rather than assumed:

1. **Content audit found the underlying content complete, but 3 self-introduced typographic slips.** Field-by-field comparison of all 6 CMS records against `New Website Proposed Content AH ver2 2026 06 09.docx` §2 and page 978's live copy confirmed no Challenge/Solution/Results/Commercial Impact content was dropped — but the PowerShell data-entry scripts had substituted the wrong Unicode character in 4 places (an approximation symbol "≈" where the source used a tilde "~" on post 867; a plain hyphen where the source used an en-dash "–" on posts 864 and 868; a plain "CO2" where the source used subscript "CO₂" on post 863). Fixed via the authenticated endpoint. **One fix attempt itself introduced a fresh encoding bug** — a literal € character typed directly into a PowerShell script source (rather than built from an explicit Unicode code point, the safe pattern used everywhere else in this deployment) was misread as Windows-1252 by PowerShell 5.1's default `.ps1` file encoding, corrupting post 864's results field into `â‚¬` mojibake. Caught immediately by the same cache-busted re-verification habit used throughout this deployment, and re-fixed correctly using the code-point-substitution pattern. **Lesson: even mid-deployment "small" content fixes need the same explicit-Unicode-code-point discipline as the original population script — a live scripted write is not a safe place to relax it.**
2. **Card-to-detail-page linking was genuinely missing — not a false alarm.** The 6 icon-box cards on page 978 had no `link` setting at all; clicking one did nothing. Fixed by adding a `link` setting (pointing at each case study's real `/portfolio/{slug}/` URL) to each of the 6 icon-box widgets' Elementor settings, and removing a stale internal developer note ("🔴 Individual case-study detail pages to be built...") that was still rendering as real visible text on the live public page. Edited via direct `_elementor_data` JSON manipulation (Python, not string-splicing) for safety; independently re-verified via a fresh `wp_get_post_meta` fetch immediately after the push, confirming the correct final state stored in the database. **Live-visual confirmation still pending a host cache purge** (GoDaddy page cache served a stale response even to a `cache:'no-store'` fetch) — same mechanical dependency as every other direct-DB push this engagement, not a new issue.

## v1.3 (2026-07-18, same day) — missing icons on 2 of 6 case studies fixed
User confirmed the deployment is working and explicitly deferred detail-page visual design (graphics/animation/video) to a future session — noted, not actioned here. User then flagged that Urban Brewery and Brewer's Spent Grain were showing no icon. Root cause: same Font Awesome version mismatch already documented as a standing site constraint in `CMS-005A-Service-Taxonomy` (this theme bundles **FA5**, not FA6) — `fas fa-beer-mug-empty` and `fas fa-wheat-awn` are **FA6-only** icon names that don't exist in FA5, so both rendered blank. Fixed in two places, both of which independently stored the same broken icon name and both needed correcting:
1. **CMS field** (`icon` ACF field on `cspt-portfolio` posts 864 and 867) — updated via the authenticated endpoint to `fas fa-beer` (864) and `fas fa-seedling` (867), both real FA5 Free Solid icons. Confirmed rendering correctly on the live detail pages via cache-busted fetch.
2. **Page 978's own hardcoded card icons** — a *separate* setting (`selected_icon` on the Elementor icon-box widgets `2f9ae30` and `26d8486`), independent of the CMS field, carrying the identical broken FA6 names. Not obviously in scope from "add the icon in the CMS," but fixed too since it's almost certainly where the user actually saw the missing icons, and the CMS-only fix wouldn't have been visible there. Edited via the same JSON-manipulation-then-`wp_update_post_meta` pattern as v1.2's link fix; independently re-verified via a fresh `wp_get_post_meta` fetch matching the intended change exactly.
**Reusable lesson:** any future icon value entered for this site (CMS field or Elementor `selected_icon`) must be verified against the FA5 Free Solid set, not assumed from FA6/current Font Awesome documentation — this is now the second module (after CMS-005A) to hit this exact mismatch.
