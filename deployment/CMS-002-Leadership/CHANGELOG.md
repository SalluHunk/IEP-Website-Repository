## v1.2 — 2026-07-18 (root cause found and fixed)
- v1.1's `register_post_meta()` fix was confirmed, via a temporary diagnostic REST endpoint, to actually register correctly (`registered_meta_key_exists()` returned true for all 7 keys) — yet writes still silently failed.
- Root cause found via a second diagnostic (an internal `rest_do_request()` write test): ACF Pro exposes its own fields under a dedicated **`"acf"`** key in REST responses, entirely separate from WordPress core's generic `"meta"` key. Writing to `meta` was never going to reach ACF field data, regardless of `register_post_meta()` state — confirmed directly, not inferred, by inspecting a real REST response showing a populated `"acf": {...}` object alongside a write to `meta` that had no effect.
- No available MCP tool can send the `acf` key directly. Fix: replaced the `register_post_meta()` approach entirely with a small authenticated custom endpoint (`/wp-json/iep-cms/v1/team-member/{id}`, GET + POST) that reads/writes via ACF's own `get_field()`/`update_field()` functions — the same mechanism ACF's own admin UI uses, so field-type formatting and the `_fieldname` reference row are handled correctly automatically.
- Endpoint requires a shared-secret header (`X-IEP-CMS-Key`) generated via `openssl rand -hex 32` — intentionally not public, since it can write content (unlike the earlier temporary diagnostics, which were read-only and have been removed in this revision).
- **Action needed:** replace the snippet's contents with this revision (the whole file again — this removes the non-working `register_post_meta()` block and the temporary diagnostics, replacing them with the working endpoint).
